# 手动验收场景

本插件包含大量确定性逻辑（正则提取、行号计算、去重、命名空间归一化等），这些都可以通过 pytest 离线验证。但以下场景涉及大语言模型（LLM）对语义、上下文和证据优先级的判断，**无法通过确定性单元测试断言**，必须在接入真实 LLM 后手动验证。

## 说明

- 每个场景只描述行为类别，不绑定具体学院、系统或案例。
- 验收时应在真实知识库中构造或找到符合描述的文档，观察 Agent 最终回复。
- 判断标准：回复必须只基于实际读取到的证据 `[E#]`，且符合对应场景的预期约束。

---

## 场景 1：语义适用性判断

**触发条件**：用户问题中的关键词在多份材料中同时出现，但材料回答的是不同主体或不同动作。

**预期行为**：
- 模型不能仅因为关键词命中就把材料当作结论依据。
- 模型应确认证据中的主体、动作、属性、修饰对象、条件和适用范围与用户问题一致。
- 若关系不一致，应继续检索或回复资料不足。

**不可断言原因**：LLM 对“主体-动作关系是否一致”的理解无法离线穷举。

---

## 场景 2：现行规则 vs 历史材料

**触发条件**：用户询问“今年、当前、现在、最新”或未指定年份的现行规则，而知识库中同时存在未标记版本、旧版或归档材料。

**预期行为**：
- 优先引用未标记旧版/归档、状态为 `current` 的材料。
- 若只有 `historical` 或 `archived` 材料，必须先说明“当前信息未找到”，再旧资料作为带年份的历史参考。
- 不能直接把历史年份的日期、流程、要求当作当前事实。

**不可断言原因**：版本元数据可以离线提取，但 LLM 是否真正按优先级引用、是否正确转述“当前未找到”需要语义判断。

---

## 场景 3：近似名称的正式对应

**触发条件**：用户使用的名称与知识库正式名称不完全一致（口语化、缩写、旧称等）。

**预期行为**：
- 模型先用用户原始名称搜索。
- 只有在知识库材料明确证明两个名称指向同一对象时，才能在回答中说明“知识库中的正式名称是××”。
- 不能仅凭文字相似或模型常识自行“纠正”用户。

**不可断言原因**：是否为“明确证明”依赖 LLM 对文本等价关系的理解，无法通过字符串匹配完全覆盖。

---

## 场景 4：残缺/截断证据

**触发条件**：`read_doc` 返回的内容恰好截断在句子中间，或上下文不足以支撑完整结论。

**预期行为**：
- 模型不得补全残句、不得编造后半句。
- 应说明资料不完整，或建议用户读取更大范围/更完整文档。

**不可断言原因**：截断位置和大模型是否“脑补”都是语义层面的判断，离线无法穷举所有截断形式。

---

## 场景 5：QA 文档块独立判断

**触发条件**：一篇 QA 文档中包含多个问答块，其中某些块标记为 `resolved`，另一些块标记为 `no_answer_found`，且它们涉及不同子主题。

**预期行为**：
- 对于有正面证据的子主题，模型应正常回答，不受相邻 `no_answer_found` 块的干扰。
- 对于只有 `no_answer_found` 的子主题，模型应说明暂无可靠资料，不能用其他块的侧面信息推断。

**不可断言原因**：块拆分可以离线验证，但 LLM 是否把 `no_answer` 当作“该文档整体无答案”而误伤其他子主题，需要真实生成验证。
